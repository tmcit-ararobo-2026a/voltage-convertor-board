## 1. ブランチ戦略

```
main
├── feat/<issue番号>-<簡潔な説明>   # 例: feat/12-add-motor-controller
└── fix/<issue番号>-<簡潔な説明>       # 例: fix/34-pwm-overflow
```

**ルール**

- `feat/` は新機能追加、`fix/` はバグ修正・不具合対応に使用します。
- ブランチ名はすべて小文字・ハイフン区切りとします。
- 1ブランチ1Issue・1機能を原則とします。
- マージ後のブランチは速やかに削除してください。

## 2. コミットメッセージ規約

[Conventional Commits](https://www.conventionalcommits.org/ja/v1.0.0/) をベースに、習得コストを下げるため5種類に絞ります。

```
<type>(<scope>): <概要（体言止め・30文字以内）>

[本文（任意）: 何を・なぜ]

[フッター（任意）]
```

**typeの一覧**

| type | 用途 |
| :--- | :--- |
| `feat` | 新機能・新モジュールの追加 |
| `fix` | バグ修正・動作不良の修正 |
| `refactor` | 動作を変えないコード改善 |
| `docs` | ドキュメント・コメントのみの変更 |
| `chore` | 環境・設定・CIの変更 |

**`feat` か `refactor` か迷った場合:** 外から見た振る舞いが変わるなら `feat`、変わらないなら `refactor`。

**記述例**

```
feat: モータコントローラノードの追加

速度指令をCANフレームに変換しSTM32へ送信するノードを実装。
既存のCANインターフェイスに依存する形で設計した。

closes #12
```

```
fix: PWMデューティ比の演算オーバーフロー修正
```

**ルール**

- 概要は体言止めで記述します（「追加した」「追加する」ではなく「追加」）
- 本文には「何を・なぜ」を記述し、「どのように」はコードに委ねます
- 関連Issueがある場合はフッターに `closes #<Issue番号>` を記載します

## 3. Issueの作成

作業着手前に必ずIssueを作成してください。

### バグ報告テンプレート

```markdown
## 概要
<!-- バグの内容を簡潔に -->

## 再現手順
1.
2.
3.

## 期待する動作

## 実際の動作

## 補足
<!-- ログ・スクリーンショット等 -->
```

### 機能追加テンプレート

```markdown
## 概要
<!-- 追加したい機能を簡潔に -->

## 背景・目的
<!-- なぜこの機能が必要か -->

## 実装方針
<!-- クラス図・データフロー等があれば添付 -->

## 完了条件
- [ ]
- [ ]
```

**ルール**

- タイトルは「`[種別] 概要`」の形式とします（例: `[feat] モータコントローラノードの追加`）
- 実装前にアーキテクチャ（クラス図・データフロー図）をIssueに添付し、合意を得てから着手してください

## 4. プルリクエストの作成

### PRテンプレート

```markdown
## 関連Issue
closes #<Issue番号>

## 変更内容
<!-- 何を・なぜ変更したか -->

## 動作確認
- [ ] シミュレーション（Isaac Sim）で確認済み
- [ ] 実機で確認済み（該当する場合）
- [ ] clang-formatによるフォーマット確認済み

## レビュー観点
<!-- レビュアーに特に見てほしい箇所 -->

## 補足
<!-- スクリーンショット・ログ等 -->
```

**ルール**

- 1名のレビュー承認をもってマージ可とします
- CIが通過していない状態でのマージは禁止します
- WIPのPRには `[WIP]` プレフィックスをタイトルに付けてください
